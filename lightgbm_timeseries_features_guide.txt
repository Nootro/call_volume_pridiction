LightGBM時系列特徴量説明資料（技術者向け）

============================================================
1. 本資料の目的
============================================================
本資料は、LightGBM を用いた時系列予測において一般的に使用される特徴量の意味を、技術者向けに整理した説明資料である。
特に、lag、曜日（day of week / dow）、rolling など、実務でほぼ必須となる特徴量群について、
「何を表すか」「なぜ効くのか」「どのような場面で有効か」「実装時の注意点」を体系的に記述する。

LightGBM は表形式データを入力とする勾配ブースティング木モデルであり、ARIMA や Prophet のように
時系列構造をモデル内部に直接持っているわけではない。
そのため、時系列に特有の情報（自己相関、季節性、イベント影響、制度変更、直近トレンドなど）は、
特徴量として明示的に与える必要がある。
言い換えると、LightGBM の時系列予測性能は「アルゴリズムそのもの」だけでなく、
どの時系列特徴量を、どの定義で、どの horizon を想定して作るかに大きく依存する。

============================================================
2. LightGBM における時系列特徴量設計の基本思想
============================================================
時系列予測では、目的変数 y_t は独立同分布ではなく、過去値・カレンダー・外生イベント・制度変更などの影響を受ける。
LightGBM はこれらを自動では認識しないため、以下のような情報を特徴量として埋め込む。

- 過去の実績値を使って自己相関を表す（lag）
- 過去数日/数週の平均や分散を使って局所トレンドを表す（rolling）
- 曜日・月・祝日などで周期性を表す（calendar features）
- 差分や比率で変化量や勢いを表す（difference / ratio）
- 制度変更や仕様変更をフラグ化してレジーム変化を表す（regime features）

この設計思想の重要点は、「予測時点で利用可能な情報だけで特徴量を作る」ことである。
時系列予測では未来実績を使うリークが極めて起きやすく、オフライン精度だけ高く本番で崩れる典型原因になる。

============================================================
3. 代表的な特徴量カテゴリ一覧
============================================================
本資料では以下のカテゴリを扱う。

1. lag 特徴量
2. seasonal lag 特徴量
3. rolling / moving statistics
4. expanding / cumulative 特徴量
5. difference / change rate 特徴量
6. calendar 特徴量（dow, month, month-end, holiday など）
7. trend / time index 特徴量
8. event / campaign / billing / special day 特徴量
9. ratio / share 特徴量
10. regime-change / policy-change 特徴量
11. interaction 特徴量
12. target encoding 的な集約特徴量の注意

============================================================
4. lag 特徴量
============================================================
4.1 定義
lag 特徴量は、過去の目的変数をそのまま説明変数として使うものである。
たとえば以下のように定義する。

- lag1  = y_(t-1)
- lag7  = y_(t-7)
- lag14 = y_(t-14)
- lag28 = y_(t-28)

4.2 意味
lag は「過去の値が未来の値に影響する」という自己相関を直接表現する。
日次データにおいて lag1 は直近日の慣性、lag7 は曜日周期、lag28 は4週周期の近似を表すことが多い。

4.3 なぜ効くのか
多くの業務時系列では、今日の件数は昨日・先週同曜日・先月近辺の状況と相関する。
LightGBM は lag を入力として与えられると、「lag7 が高い日で、かつ平日なら予測値を高める」といった
条件分岐を学習できる。

4.4 コールセンター日次予測での解釈例
- lag1: 前日の入電件数の影響
- lag7: 先週同曜日の入電件数の影響
- lag14: 2週前同曜日の影響
- lag28: 月次に近い周期や4週間前の運用状態の影響

4.5 実装上の注意
- 予測対象日 t に対して、lag は必ず t より前の実績だけを使う
- 学習時と推論時で同じ定義にする
- horizon が長くなると使える lag の価値が下がる場合がある
- recursive forecasting では、将来日の lag を予測値で埋める必要があり、誤差伝搬が起きる

4.6 実務上のポイント
日次予測では lag7, lag14, lag28 は非常によく使われる。
一方で、lag1 や lag2 は短期ショックに強いが、ノイズも拾いやすいため、業務特性に応じて採否を決める。

============================================================
5. seasonal lag 特徴量
============================================================
5.1 定義
seasonal lag は、単なる過去値ではなく「周期を意識した過去値」である。
日次系列では特に以下が多い。

- same_day_last_week  = y_(t-7)
- same_day_last_2week = y_(t-14)
- same_day_last_4week = y_(t-28)

月次系列なら y_(t-12)、時間単位なら y_(t-24), y_(t-168) などが相当する。

5.2 意味
時系列に周期性がある場合、同じ周期位置の過去値は強い説明力を持つ。
コールセンターでは「月曜は月曜らしい」「月末は月末らしい」という構造があり、
seasonal lag はそれをシンプルに保持する。

5.3 注意点
季節構造が制度変更や大型施策で崩れている場合、seasonal lag の信頼性は落ちる。
そのため、仕様変更後の立ち上がり局面では、regime-change flag と併用して解釈する必要がある。

============================================================
6. rolling 特徴量
============================================================
6.1 定義
rolling 特徴量は、直近 N 期間の平均・合計・最大・最小・標準偏差などを表す。
例:

- rolling_mean_7   = mean(y_(t-7) ... y_(t-1))
- rolling_mean_28  = mean(y_(t-28) ... y_(t-1))
- rolling_std_7    = std(y_(t-7) ... y_(t-1))
- rolling_max_14   = max(y_(t-14) ... y_(t-1))
- rolling_min_14   = min(y_(t-14) ... y_(t-1))

6.2 意味
rolling は「局所的な水準」「最近の安定度」「直近の変動の大きさ」を表す。
lag が一点の過去値を見るのに対し、rolling は一定区間の統計量を見る。

6.3 なぜ効くのか
実務系列では、単一日の値よりも「直近1週間の平均水準」の方が安定的な説明変数になる場合が多い。
また、rolling_std は変動が大きい局面かどうかを示すため、予測区間の不安定さを間接的に表せる。

6.4 コールセンターでの解釈例
- rolling_mean_7: 直近1週間の平均入電水準
- rolling_mean_28: 直近4週間の基準水準
- rolling_std_7: 直近1週間の変動の激しさ
- rolling_max_14: 直近2週間で最も忙しかった日の水準

6.5 実装上の注意
- rolling の窓に未来日を含めない
- 学習時に中央寄せや未来参照の集計をしない
- 欠損がある場合の扱いを統一する
- 窓長は業務周期に対応させる（7日, 14日, 28日など）

6.6 よくある失敗
- t 日予測に t 日を含む rolling_mean を使ってしまう
- 検証時だけ簡略化した rolling を使い、本番と不一致になる
- 祝日や異常値が多い区間で rolling_mean を過信する

============================================================
7. expanding / cumulative 特徴量
============================================================
7.1 定義
expanding は「系列開始から現時点直前まで」の累積統計量である。
例:

- expanding_mean = mean(y_1 ... y_(t-1))
- expanding_sum  = sum(y_1 ... y_(t-1))
- cumulative_count_of_holiday_before_t

7.2 意味
短期水準ではなく、より長期の平均的なベースラインを表す。
rolling が局所、expanding が長期基準という関係で理解するとよい。

7.3 使いどころ
- 長期的な平均水準との差を見るとき
- 学習期間全体に対して現在が高いか低いかを判断したいとき
- データ量が十分あり、長期トレンドが比較的安定しているとき

7.4 注意点
制度変更をまたぐ系列では expanding が旧 regime を強く含むため、
仕様変更後の振る舞いを鈍らせる可能性がある。
この場合、変更前後で expanding を分ける、または post_change 後だけの expanding を作る工夫が有効である。

============================================================
8. difference / 変化量特徴量
============================================================
8.1 定義
difference は、過去との差分である。
例:

- diff_1  = y_(t-1) - y_(t-2)
- diff_7  = y_(t-1) - y_(t-8)
- rolling_mean_gap_7_28 = rolling_mean_7 - rolling_mean_28

比率版としては以下もよく使う。
- ratio_7_28 = rolling_mean_7 / rolling_mean_28
- yoy_like_ratio など

8.2 意味
差分や比率は「水準」ではなく「勢い」を表す。
直近で増加傾向か、下降傾向か、短期水準が長期基準より高いかを捉えられる。

8.3 コールセンターでの解釈
- rolling_mean_7 が rolling_mean_28 より高い: 直近の問い合わせが基準より増えている
- diff_7 が大きい: 先週よりも足元で増減が起きている

8.4 注意点
差分はノイズを増幅しやすい。
また、元系列にゼロや極小値が多い場合、比率系は不安定になるため、分母の下限処理が必要である。

============================================================
9. calendar 特徴量
============================================================
9.1 day of week（dow）
- 例: dow = 0..6
- 意味: 曜日ごとの周期性を表す
- なぜ効くか: コールセンターは月曜集中、土日低下など、曜日効果が極めて強いことが多い
- 実装注意: 数値の大小関係として扱わせたくない場合、one-hot や category として扱うことを検討する

9.2 is_weekend / is_business_day
- 意味: 平日/休日の大枠差分を表す
- dow より粗いが頑健
- 土日祝で問い合わせ行動が大きく異なる業務では重要

9.3 month
- 意味: 月ごとの季節性、年度イベント、請求・契約更新などの月次構造を表す
- 注意: month 単体では序数的意味が弱いので、カテゴリ的に扱うか cyclic encoding を検討する

9.4 day_of_month / is_month_start / is_month_end
- 意味: 月初・月末・給料日・締め日などの業務イベント近辺の影響を表す
- コールセンターでは請求確定や締め日付近で問い合わせが偏ることがある

9.5 holiday / pre_holiday / post_holiday
- 意味: 祝日当日だけでなく、前後日の需要変化を表す
- 特に大型連休前後は通常の dow だけでは表現不足になる

9.6 week_of_year / quarter / fiscal_period
- 年間のどの位置かを示す
- 繁忙期・閑散期がある場合に有効

============================================================
10. trend / 時間インデックス特徴量
============================================================
10.1 定義例
- time_idx = 連番
- days_from_start
- weeks_from_start

10.2 意味
時系列全体のゆるやかな増減トレンドを表す。
Prophet では trend がモデル内部にあるが、LightGBM では必要に応じて time index を特徴量として与える。

10.3 注意点
木モデルは時間インデックスから単純な線形外挿を得意としないため、
trend feature は「補助的」には有効だが、長期予測を単独で支えるほどではない。
急激な制度変更がある場合は、単純な time_idx より regime flag の方が重要になる。

============================================================
11. event / campaign / billing / special day 特徴量
============================================================
11.1 定義
業務固有のイベントをフラグやイベントまでの日数として与える。
例:
- is_campaign_period
- days_to_billing_date
- is_system_maintenance_day
- is_rate_plan_change_week

11.2 意味
一般的な calendar では表現できない業務イベントの影響を表す。
コールセンターでは料金改定、障害、キャンペーン、端末発売日などが件数を大きく動かし得る。

11.3 注意点
イベントマスタの欠損や更新漏れがあると、本番で特徴量再現性が崩れる。
この種の特徴量は効果が大きい一方、運用依存性も強い。

============================================================
12. ratio / share 特徴量
============================================================
12.1 定義
総量に対するカテゴリ比率、あるいは構成比のラグや rolling を使う。
例:
- share_catA = catA / total
- share_catA_lag7
- share_catA_rolling_mean_14

12.2 意味
カテゴリ件数そのものではなく、全体の中でどのカテゴリがどれだけの割合を占めるかを表す。
総量変動とカテゴリ配分変動を分離したい場合に有効。

12.3 コールセンターでの有効性
仕様変更で「その他」から他カテゴリへ流れるようなケースでは、
件数そのものより share の方が変化の本質を捉えやすい。

12.4 注意点
share は total が小さい日に不安定になりやすい。
また、予測時点で total の将来値が不明なら、
(1) total を別モデルで予測する、(2) share 自体を直接予測する、という設計が必要になる。

============================================================
13. regime-change / 仕様変更特徴量
============================================================
13.1 定義例
- post_change: 仕様変更後なら 1
- days_since_change: 変更後経過日数
- before_after_version
- routing_rule_version

13.2 意味
制度変更、業務ルール変更、分類基準変更、システム更改などで
データ生成過程が変わったことを明示的に表す。

13.3 コールセンターでの重要性
問い合わせ内容そのものは同じでも、振り分け方式が変わるとカテゴリ件数分布は変わる。
このとき regime-change 特徴量がないと、モデルは「突然説明できない変化が起きた」と見なして不安定になる。

13.4 実装上の注意
- 変更日が業務上いつから有効かを厳密に定義する
- 変更後の立ち上がり期間を days_since_change で滑らかに表現する
- 旧データと新データを同一分布と仮定しない

============================================================
14. interaction 特徴量
============================================================
14.1 例
- dow × post_change
- is_month_end × category_share_lag7
- holiday × rolling_mean_7

14.2 意味
「曜日効果は制度変更前後で異なる」「月末効果は最近の水準によって強さが変わる」といった
条件付きの関係を表現する。

14.3 LightGBM における位置づけ
木モデルはある程度自動で交互作用を学習できるため、
明示的 interaction は必須ではない。
ただし、業務仮説が明確な場合は追加する価値がある。

============================================================
15. horizon が延びたときの特徴量の扱い
============================================================
1か月先予測と3か月先予測では、特徴量の有効性が変わる。

- lag / rolling: 短期では強いが、長期では使いにくくなる
- calendar / holiday: 長期でも比較的安定して使える
- regime-change: 変更直後は重要、安定後は相対的重要度が下がることがある
- trend: 長期では補助的に効くが、外挿能力には限界がある

このため、短期予測で LightGBM が強くても、長期予測では Prophet や foundation model が相対的に有利になる場合がある。

============================================================
16. Prophet と LightGBM の違い
============================================================
Prophet はトレンド・季節性・祝日効果をモデル内部で表現する統計モデルであり、
calendar 系の構造を自然に扱える。
一方 LightGBM は、lag、rolling、dow、holiday、regime flag などを外から与えることで
柔軟な非線形予測を行う。

整理すると以下の通り。
- Prophet: 構造を内包する、長め horizon に比較的安定、解釈しやすい
- LightGBM: 特徴量次第で高精度、外生変数や複雑な非線形に強い、ただし設計依存が大きい

============================================================
17. 実装チェックリスト
============================================================
[特徴量定義]
- lag はすべて過去参照のみか
- rolling に未来日が含まれていないか
- holiday マスタは将来日まで整備されているか
- regime-change 日付は業務定義と一致しているか
- category / dow のエンコーディング方法は統一されているか

[学習・検証]
- 特徴量生成を fold ごとに作り直しているか
- 本番と同じ情報制約で validation しているか
- recursive forecast 時の lag 更新方法が本番と一致しているか

[運用]
- イベントマスタの更新責任者が決まっているか
- 欠損発生時の補完ルールがあるか
- 仕様変更時に regime flag を追加する運用があるか

============================================================
18. よくある失敗例
============================================================
1. 未来実績を含む rolling を使ってしまう
2. 学習時は真の lag、推論時は予測 lag になる差を無視する
3. 仕様変更を特徴量化せず、旧データと新データをそのまま混ぜる
4. dow や holiday を入れず、lag だけに依存する
5. 小カテゴリで比率特徴量を使い、分母ゼロ問題を放置する
6. オフラインでだけ高精度な特徴量を採用してしまう

============================================================
19. そのまま転記できる要約
============================================================
LightGBM による時系列予測では、モデルが時系列構造を自動理解するわけではないため、
過去実績を表す lag、直近水準を表す rolling、曜日・祝日・月末などの calendar 特徴量、
制度変更を表す regime-change 特徴量を明示的に与える必要がある。
lag は自己相関、rolling は局所水準と変動、dow は曜日周期、holiday は特異日効果、
share はカテゴリ配分、regime flag は仕様変更の影響を表す。
重要なのは、これらの特徴量を必ず予測時点で利用可能な情報だけで作ることであり、
rolling や lag のリーク、仕様変更の未反映、horizon 拡大時の特徴量価値低下に注意する必要がある。
