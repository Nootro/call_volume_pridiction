時系列予測におけるモデル分類の技術解説
統計モデル / 機械学習モデル / 時系列基盤モデル
Prophet / LightGBM / TimesFM / Chronos を含む実務整理版

======================================================================
1. 文書の目的
======================================================================

本テキストは、時系列予測に用いられる主要なモデル群を、技術者向けに整理して説明することを目的とする。
特に、実務でよく議論になる以下の3系統を区別して理解できるようにする。

  1) 統計モデル
  2) 機械学習モデル
  3) 時系列基盤モデル（Time Series Foundation Models）

あわせて、代表例として以下の4つを具体的に扱う。

  ・Prophet
  ・LightGBM
  ・TimesFM
  ・Chronos

この文書では、単なる定義紹介ではなく、
  ・何を得意とするのか
  ・どのような入力を必要とするのか
  ・どのようなケースで使いやすいのか
  ・どのような弱点があるのか
を実務観点でまとめる。

======================================================================
2. 時系列予測とは何か
======================================================================

時系列予測とは、過去から現在までの時系列データをもとに、将来の値を予測することである。
Google Cloud は、時系列予測を「過去の時系列データをモデル化して将来時点を予測する統計的または機械学習的アプローチ」と説明している。
参照: https://cloud.google.com/learn/what-is-time-series

時系列予測が通常の回帰と異なるのは、データに時間順序があり、
  ・過去が未来に影響する
  ・曜日や季節などの周期がある
  ・トレンドが変化する
  ・イベントや制度変更で不連続な変化が起きる
という点にある。

Google Cloud も、時系列予測には seasonality、holiday effects、data sparsity、changing trends などの固有課題があり、
通常の機械学習でよく使う k-fold cross validation は data leakage を起こしやすいと説明している。
参照: https://cloud.google.com/learn/what-is-time-series

このため、時系列予測ではモデルの種類だけでなく、
  ・時間順序を守った検証
  ・本番時点で使える情報だけでの特徴量生成
  ・将来情報リークの防止
が必須になる。

======================================================================
3. モデルの大分類
======================================================================

時系列予測のモデルは、大きく以下の3系統に分けて理解すると整理しやすい。

---------------------------------------------------------------------
3.1 統計モデル
---------------------------------------------------------------------

定義:
  時系列データのトレンド、季節性、自己相関、祝日効果などを、比較的明示的な数理構造で表現するモデル群。

代表例:
  ・ARIMA
  ・SARIMA
  ・指数平滑法
  ・ETS
  ・Prophet

特徴:
  ・構造が比較的解釈しやすい
  ・トレンドや季節性を直接扱いやすい
  ・長期の基礎的な動きを表現しやすい
  ・説明変数が少なくても動く

強み:
  ・曜日性、季節性、トレンドが強い系列に向く
  ・データサイエンス以外の関係者にも説明しやすい
  ・ベースラインとして優秀なことが多い

弱み:
  ・多数の外生変数や複雑な非線形相互作用を扱うのは苦手
  ・カテゴリ定義変更や制度変更のような不連続な変化に弱いことがある
  ・細かな短期変動や複雑な条件分岐は拾いにくい

---------------------------------------------------------------------
3.2 機械学習モデル
---------------------------------------------------------------------

定義:
  時系列を直接数式分解するというより、過去実績、曜日、祝日、イベント、lag、rolling などを特徴量として作り、
  それらから未来値を予測するモデル群。

代表例:
  ・LightGBM
  ・XGBoost
  ・Random Forest
  ・ニューラルネット系回帰モデル

特徴:
  ・特徴量設計に強く依存する
  ・非線形性や相互作用を学習しやすい
  ・多様な外生変数を柔軟に扱える

強み:
  ・曜日×祝日×イベントのような複雑な相互作用を表現しやすい
  ・カテゴリ配分や短期補正に強い
  ・特徴量重要度を見ながら改善しやすい

弱み:
  ・特徴量リークに非常に弱い
  ・長期トレンドの外挿は苦手
  ・本番時点で使えない特徴量を学習に混ぜると、オフラインだけ良く見える

---------------------------------------------------------------------
3.3 時系列基盤モデル
---------------------------------------------------------------------

定義:
  大量かつ多様な時系列データで事前学習された汎用モデルを用い、ゼロショットまたは少量追加データで予測するモデル群。

代表例:
  ・TimesFM
  ・Chronos
  ・TimeGPT（本資料では参考名のみ）

特徴:
  ・事前学習済みモデルとして利用できる
  ・系列ごとに一から学習しなくても、ある程度の予測性能を出せる
  ・ゼロショットまたは few-shot で使えることが大きな特徴

強み:
  ・データが少ない系列でもベースラインを素早く作れる
  ・多様なドメインにまたがる汎化性能が期待できる
  ・新規系列や大量系列の比較検証に向く

弱み:
  ・ドメイン特有の業務ルールや特殊な制約はそのままでは反映しにくい
  ・内部構造の解釈は統計モデルより難しい
  ・ゼロショット性能が高くても、業務要件に最適とは限らない
  ・独自特徴量を大量投入する従来型 ML ほど柔軟ではない場合がある

4. Prophet の技術的整理
======================================================================

Prophet は Meta のオープンソース時系列予測手法であり、公式サイトでは
「非線形トレンドを yearly, weekly, daily seasonality と holiday effects とともに additive model で表現する forecasting procedure」
と説明されている。
参照: http://facebook.github.io/prophet/

公式サイトの要点:
  ・Prophet is a procedure for forecasting time series data based on an additive model
  ・non-linear trends are fit with yearly, weekly, and daily seasonality, plus holiday effects
  ・robust to missing data and shifts in the trend
  ・typically handles outliers well
  参照: http://facebook.github.io/prophet/

---------------------------------------------------------------------
4.1 Prophet の考え方
---------------------------------------------------------------------

Prophet は、系列を大まかに
  ・トレンド
  ・季節性
  ・祝日効果
  ・誤差
に分解して予測する。

そのため、
  ・需要の長期増減
  ・曜日性
  ・年次季節性
  ・祝日効果
のような構造を持つ系列に向いている。

---------------------------------------------------------------------
4.2 Prophet の強み
---------------------------------------------------------------------

  ・トレンドと季節性を自然に扱える
  ・祝日を入れやすい
  ・欠損や外れ値に比較的頑健
  ・長めの予測でも比較的安定しやすい
  ・モデルの説明がしやすい

実務的には、
  ・総量予測
  ・安定カテゴリの予測
  ・ベースライン作成
に向いていることが多い。

---------------------------------------------------------------------
4.3 Prophet の弱み
---------------------------------------------------------------------

  ・複雑な非線形相互作用は苦手
  ・多くの外生変数を柔軟に扱う用途には限界がある
  ・カテゴリ配分の急変には追随しづらい
  ・制度変更やラベル定義変更に対して、意味変化を直接は理解しない

たとえば、分類ルールが変わって「その他」が他カテゴリへ再配分された場合、
Prophet から見ると急なレベルシフトやトレンド変化にしか見えない。
そのため、カテゴリ直予測では不利になるケースがある。

---------------------------------------------------------------------
4.4 Prophet が向くケース
---------------------------------------------------------------------

  ・曜日性や祝日性が強い
  ・年次季節性がある
  ・長期トレンドを素直に見たい
  ・解釈性を重視したい
  ・まず強いベースラインを作りたい

======================================================================
5. LightGBM の技術的整理
======================================================================

LightGBM は Microsoft が公開している gradient boosting framework であり、
公式ドキュメントでは「tree based learning algorithms を使う gradient boosting framework」と説明されている。
参照: https://lightgbm.readthedocs.io/en/latest/

公式ドキュメントの要点:
  ・tree based learning algorithms
  ・faster training speed and higher efficiency
  ・lower memory usage
  ・better accuracy
  ・parallel, distributed, and GPU learning support
  ・capable of handling large-scale data
  参照: https://lightgbm.readthedocs.io/en/latest/

---------------------------------------------------------------------
5.1 LightGBM の考え方
---------------------------------------------------------------------

LightGBM 自体は時系列モデルではない。
そのため、時系列予測で使うには、時系列を特徴量へ変換する必要がある。

典型的な特徴量:
  ・曜日
  ・祝日
  ・月初 / 月末
  ・lag_7, lag_14, lag_28
  ・rolling_mean_7, rolling_std_28
  ・イベントフラグ
  ・カテゴリシェア履歴

つまり LightGBM は、
  「過去実績やカレンダー情報を特徴量化した tabular problem」
として時系列予測を解くイメージである。

---------------------------------------------------------------------
5.2 LightGBM の強み
---------------------------------------------------------------------

  ・非線形な相互作用を学習できる
  ・多数の特徴量を柔軟に使える
  ・曜日×祝日×イベントのような複雑な条件を表現しやすい
  ・カテゴリ配分や短期補正に向く
  ・特徴量重要度で改善の方向を見やすい

たとえばコールセンターでは、
  ・月末かつ平日で、さらに請求通知後である
  ・祝日翌日で、かつ大型連休明けである
といった複雑な条件が呼量に効くことがある。
このような条件は LightGBM が得意とする。

---------------------------------------------------------------------
5.3 LightGBM の弱み
---------------------------------------------------------------------

  ・特徴量設計ミスが精度に直結する
  ・未来情報リークのリスクが高い
  ・長期トレンド外挿は苦手
  ・ホライズンが長くなると lag 系が使いにくくなる
  ・本番で利用できない特徴量を学習に使うと崩壊する

たとえば、2月末時点で5月を予測するのに、4月実績を使って rolling 特徴量を作れば、
オフラインでは高精度でも本番では不可能な予測になる。

---------------------------------------------------------------------
5.4 LightGBM が向くケース
---------------------------------------------------------------------

  ・短中期予測
  ・イベント影響が大きい
  ・カテゴリ配分予測
  ・多数の外部要因を使いたい
  ・非線形補正が必要

6. TimesFM の技術的整理
======================================================================

TimesFM は Google Research の時系列基盤モデルであり、
Google Research Blog では「a decoder-only foundation model for time-series forecasting」と説明されている。
参照: https://research.google/blog/a-decoder-only-foundation-model-for-time-series-forecasting/

Google Research Blog の要点:
  ・100 billion real world time-points で事前学習
  ・200M parameters の decoder-only foundation model
  ・zero-shot performance が強い
  ・unseen datasets of different domains and temporal granularities でも高性能
  ・time-point の contiguous group を patch として token 化して transformer に入力
  参照: https://research.google/blog/a-decoder-only-foundation-model-for-time-series-forecasting/

Google Cloud BigQuery の公式ドキュメントでも、TimesFM は
  ・Google Research の open source TimesFM の実装
  ・many real-world datasets の billions of time-points で pre-trained
  ・new forecasting datasets across many domains に適用できる foundation model
と説明されている。
参照: https://docs.cloud.google.com/bigquery/docs/timesfm-model

---------------------------------------------------------------------
6.1 TimesFM の考え方
---------------------------------------------------------------------

TimesFM は、自然言語処理における大規模事前学習モデルに近い発想を、時系列へ適用したものと考えると理解しやすい。

従来の統計モデルや機械学習モデルでは、系列ごとに特徴量設計や再学習が必要なことが多い。
一方 TimesFM は、大量の多様な時系列で事前学習済みのモデルを使って、
新しい系列に対してもゼロショットまたは少ない追加作業で予測する。

---------------------------------------------------------------------
6.2 TimesFM の強み
---------------------------------------------------------------------

  ・ゼロショットで多様な系列に使える
  ・事前学習済みなのでベースライン構築が速い
  ・多様なドメインや粒度にまたがる汎化性能がある
  ・長い手作業特徴量設計なしでも一定精度を期待できる

Google Cloud では、BigQuery の AI.FORECAST から TimesFM を利用でき、
独自モデル作成や管理をせず forecasting できる点を利点としている。
参照: https://docs.cloud.google.com/bigquery/docs/timesfm-model

---------------------------------------------------------------------
6.3 TimesFM の弱み
---------------------------------------------------------------------

  ・業務固有のルールや特殊イベントをそのまま細かく反映するのは難しい
  ・解釈性は Prophet より低い
  ・細かなドメイン知識を入れ込む自由度は LightGBM より低いことがある
  ・ユニバリアント前提の使い方では共変量活用が限定される場合がある

実務では、
  ・まず TimesFM を強い汎用ベースラインとして置く
  ・その上で Prophet / LightGBM と比較する
という使い方が現実的である。

---------------------------------------------------------------------
6.4 TimesFM が向くケース
---------------------------------------------------------------------

  ・大量系列の初期ベースライン作成
  ・系列ごとの特徴量設計コストを減らしたい
  ・学習データが少ない新規系列
  ・ゼロショット性能を活かした比較検証

7. Chronos の技術的整理
======================================================================

Chronos は Amazon Science の時系列基盤モデル群であり、公式 GitHub では
「The original Chronos family is based on language model architectures」と説明されている。
参照: https://github.com/amazon-science/chronos-forecasting

公式 GitHub の要点:
  ・time series is transformed into a sequence of tokens via scaling and quantization
  ・language model is trained on these tokens using cross-entropy loss
  ・probabilistic forecasts are obtained by sampling multiple future trajectories
  ・Chronos-2 は zero-shot support for univariate, multivariate, and covariate-informed forecasting tasks
  ・Chronos-2 は exogenous features を含むタスクで大きく改善
  参照: https://github.com/amazon-science/chronos-forecasting

---------------------------------------------------------------------
7.1 Chronos の考え方
---------------------------------------------------------------------

Chronos は、時系列を token sequence に変換し、言語モデルのように学習するという発想を取る。
この点で、時系列を「時間軸上の値の並び」ではなく、「言語モデルが扱える token 列」として表現するのが特徴である。

Chronos は単一点予測ではなく、複数の将来 trajectory をサンプリングして probabilistic forecasts を得るという考え方を持つ。
そのため、不確実性を扱う文脈とも相性がよい。

---------------------------------------------------------------------
7.2 Chronos の強み
---------------------------------------------------------------------

  ・ゼロショットで利用できる
  ・確率的予測に向く
  ・Chronos-2 は univariate, multivariate, covariate-informed forecasting をサポート
  ・外生変数つきタスクで改善が大きい
  ・多様な系列に対して事前学習済み知識を持つ

特に Chronos-2 の改善点として、公式 GitHub は
  ・multivariate
  ・covariate-informed forecasting
  ・state-of-the-art zero-shot performance
を挙げている。
参照: https://github.com/amazon-science/chronos-forecasting

---------------------------------------------------------------------
7.3 Chronos の弱み
---------------------------------------------------------------------

  ・解釈性は統計モデルより低い
  ・業務固有のロジックや制約は追加設計が必要
  ・確率予測をどう業務意思決定へつなぐか整理が必要
  ・ゼロショットで高性能でも、そのまま業務最適とは限らない

---------------------------------------------------------------------
7.4 Chronos が向くケース
---------------------------------------------------------------------

  ・確率的予測を重視したい
  ・多変量 / 外生変数つき基盤モデルを試したい
  ・大量系列へ汎用モデルを当てたい
  ・既存統計モデル / ML モデルの比較対象が欲しい

8. 4モデルの比較整理
======================================================================

---------------------------------------------------------------------
8.1 Prophet
---------------------------------------------------------------------

立ち位置:
  統計モデル寄り

得意:
  ・トレンド
  ・曜日性
  ・季節性
  ・祝日
  ・解釈性

苦手:
  ・複雑な非線形相互作用
  ・カテゴリ配分の急変
  ・制度変更の局所吸収

向く用途:
  ・総量予測
  ・安定系列
  ・ベースライン

---------------------------------------------------------------------
8.2 LightGBM
---------------------------------------------------------------------

立ち位置:
  機械学習モデル

得意:
  ・非線形相互作用
  ・イベント影響
  ・lag / rolling 利用
  ・カテゴリ配分

苦手:
  ・長期外挿
  ・未来リークに弱い
  ・特徴量依存が強い

向く用途:
  ・短中期予測
  ・カテゴリ配分
  ・外部要因を多く使う予測

---------------------------------------------------------------------
8.3 TimesFM
---------------------------------------------------------------------

立ち位置:
  時系列基盤モデル

得意:
  ・ゼロショット予測
  ・多ドメイン汎化
  ・ベースラインを速く作る

苦手:
  ・業務固有の細かいルール反映
  ・解釈性

向く用途:
  ・初期比較
  ・大量系列
  ・データが少ない系列

---------------------------------------------------------------------
8.4 Chronos
---------------------------------------------------------------------

立ち位置:
  時系列基盤モデル

得意:
  ・ゼロショット
  ・確率的予測
  ・Chronos-2 では multivariate / covariate-informed forecasting

苦手:
  ・運用要件への落とし込みには追加設計が必要
  ・解釈性が低い

向く用途:
  ・不確実性込みの予測
  ・基盤モデル比較
  ・外生変数つき基盤モデル検証

9. 実務での使い分け方
======================================================================

実務では、これらを対立概念としてではなく、役割の違う道具として使い分けるのがよい。

典型的な整理:

  ・Prophet
      総量の構造理解、長期トレンド、曜日性、祝日性のベースライン

  ・LightGBM
      カテゴリ配分、短期補正、イベント反応、非線形相互作用

  ・TimesFM
      ゼロショットでの汎用ベースライン比較

  ・Chronos
      ゼロショット / 確率的 / 多変量・外生変数つき基盤モデル比較

たとえばコールセンター予測であれば、
  ・総量 = Prophet
  ・カテゴリ配分 = LightGBM
  ・比較用ベースライン = TimesFM / Chronos
という設計はかなり自然である。

======================================================================
10. モデル選定で重要なこと
======================================================================

どのモデルが優れているかは、モデルの新しさではなく、予測対象と検証設計次第で変わる。

重要な観点:
  1) 予測対象は総量か、カテゴリ配分か
  2) 予測ホライズンは短いか、長いか
  3) 外部要因を多く使いたいか
  4) データ量は十分か
  5) 解釈性が必要か
  6) ゼロショット性能を重視するか
  7) 本番時点で利用可能な情報だけで評価しているか

特に最後が重要である。
統計モデルでも、機械学習モデルでも、基盤モデルでも、
未来情報を含んだ評価をすると本番では再現しない。

======================================================================
11. まとめ
======================================================================

  ・統計モデルは、トレンドや季節性を明示的に捉えるのが得意で、Prophet が代表例である
  ・機械学習モデルは、特徴量ベースで非線形性を吸収するのが得意で、LightGBM が代表例である
  ・時系列基盤モデルは、大量の時系列で事前学習された汎用モデルで、TimesFM や Chronos が代表例である
  ・Prophet は総量や安定系列、LightGBM は配分や短期補正に向くことが多い
  ・TimesFM と Chronos は、強い汎用ベースラインやゼロショット比較対象として有力である
  ・どのモデルを選ぶにしても、本番時点で使える情報だけで検証することが最優先である
