# call_volume_pridiction
現在は、2025年2月までのデータを用いて2025年3月・4月を予測する前提で、
パラメータ探索を「学習期間: 2024年12月まで、検証期間: 2025年1月～2月」として実施し、
その中で精度の良かったパラメータを使って最終的に2025年2月までで再学習し、3月・4月を予測しています。

これを3月・4月・5月予測に変更する場合、最終学習自体を大きく変える必要は必ずしもありません。
つまり、最終的に2025年2月までのデータで学習し、2025年3月～5月を予測する進め方自体は自然です。

ただし、見直すべきなのはパラメータ探索と検証設計です。
これまでの設定は2か月先予測に合わせた検証になっているため、そのままでは3か月先を含む本番条件を十分に再現できません。
3か月予測を行うのであれば、検証期間も3か月先まで含めた形に変更し、その条件でLightGBMとProphetを再比較する方が妥当です。

特にLightGBMは、lag特徴量やrolling特徴量を多く使う場合、予測ホライズンが長くなるほど不利になりやすいです。
2月末時点で5月を予測する場合、4月や5月の実績はまだ存在しないため、5月予測では使える直近実績特徴量が減り、オフラインで良く見えてもオンラインで崩れる可能性があります。
一方でProphetは、曜日性・祝日・季節性などのカレンダー要因を中心に予測するため、長めの予測期間では相対的に安定しやすい場合があります。

そのため、3月・4月ではLightGBMの方が精度が良かったとしても、3月・4月・5月まで評価対象を広げるとProphetの方が良くなることは十分にあり得ます。
これはモデルがおかしいのではなく、評価対象が2か月先予測から3か月先予測へ変わり、必要な性質が変わるためです。

したがって、2か月先予測で選ばれた最適モデルをそのまま3か月予測に流用するのではなく、
3か月先予測用の検証設計で、カテゴリごとにLightGBMとProphetをあらためて比較し直すべきです。

さらに短く一言で言うなら
最終学習を2025年2月までで行う点はそのままでよいが、3・4・5月予測に変えるなら、パラメータ探索と検証は3か月先前提に見直すべきであり、その結果として3・4月ではLightGBMが良くても、3・4・5月ではProphetが勝つことは十分あり得る。


予測対象を2か月先から3か月先へ拡張する場合、最終学習時点そのものを必ずしも変更する必要はない。
たとえば、最終的に2025年2月末までのデータで再学習し、2025年3月～5月を予測する進め方自体は自然である。
一方で、パラメータ探索およびモデル選定のためのCV設計は、予測ホライズンの変更に合わせて見直す必要がある。

従来の「学習: ～2024年12月、検証: 2025年1～2月」という構成は、2か月先予測に対しては整合的だが、3か月先予測の本番条件を十分には再現しない。
3か月先予測を行うのであれば、CVも3か月先までを含む形に変更し、その条件で各モデルを再比較するのが妥当である。

モデル特性の観点では、Prophetは曜日性・祝日・季節性などのカレンダー要因を主に利用するため、長めの予測ホライズンでも比較的安定しやすい。
一方、LightGBMはlag特徴量やrolling特徴量を活用できる短中期予測では強いが、ホライズンが伸びるほど利用可能な直近実績特徴量が減りやすく、recursive予測時の誤差伝播や未来情報リークにも注意が必要である。

そのため、2か月先評価ではLightGBMが優位であっても、3か月先まで含めた評価ではProphetが逆転することは十分にあり得る。
これはモデル性能の矛盾ではなく、評価対象のホライズンが変わることで、必要とされるモデル特性が変化するためである。

したがって、2か月先用に選定した最適モデルをそのまま3か月先予測へ流用するのではなく、
3か月先前提のCV設計に切り替えた上で、カテゴリ別・horizon別・月別にLightGBMとProphetを再比較し、必要に応じてモデルや特徴量設計を見直すべきである。
